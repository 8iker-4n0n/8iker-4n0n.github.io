## Overview

Public MkDocs-based security research site hosted on GitHub Pages. Handle: `8iker_4n0n`. Stack: MkDocs 1.6.1 + Material 9.6 (already installed in `python_env`).

Content types: IR reports, malware/sample analysis, CVE writeups, tool releases. Organized by type, not date. Tools section links out to GitHub repos rather than embedding code.

## Frontmatter Template

Every report (IR, malware, CVE) uses this YAML frontmatter:

```yaml
---
title: "Report Title"
date: YYYY-MM-DD
author: 8iker_4n0n
category: ir-report        # ir-report | malware | cve
severity: high             # critical | high | medium | low
mitre_tags:
  - T1505.003
  - T1027
ioc_summary:
  - "SHA256: abc123..."
  - "URL: http://..."
  - "IP: 1.2.3.4"
tags:
  - webshell
  - polymorphic
  - T1505.003
---
```

`tags` field powers Material's tags plugin — readers click a tag and see all reports sharing it. MITRE IDs appear in both `mitre_tags` (structured data) and `tags` (filterable index).

## Report Body Structure

Convention (not enforced by tooling) — every report follows this section order:

| Section | Purpose |
|---|---|
| `## Overview` | What happened, when, impact |
| `## Timeline` | Chronological key events |
| `## Technical Analysis` | Code, screenshots, behavior breakdown |
| `## MITRE ATT&CK Mapping` | Table: Tactic / Technique ID / Name / Notes |
| `## Indicators of Compromise` | Collapsible blocks per IOC type |
| `## Remediation` | Actions taken / recommended |
| `## References` | CVEs, vendor advisories, external links |

Use `!!! danger "IOCs"` admonition for high-visibility callouts. Use collapsible `??? "Hash list"` blocks for long IOC lists to keep pages scannable.
